# Go 社内公開前チェック

## 目的

Goで作成したツールを社内へ公開・配布する前に、リポジトリと生成物を機械的・目視で確認し、次を明確にする。

- 公開可否: `PASS` / `CONDITIONAL PASS` / `FAIL`
- 公開を止める問題
- 公開前に直す問題
- 公開後の改善事項
- 実行した検査、未実施の検査、判断根拠

このスキルは、コードを無断で修正したり公開したりしない。検査を基本とし、修正は利用者から明示的に依頼された場合だけ行う。

## 想定する対象

- Go製CLI、バッチ、常駐プロセス、社内Webサービス
- 社内で再利用するGoライブラリ
- Windows向け `.exe`、Linux向けバイナリ、コンテナイメージ
- GitHub、GitHub Enterprise、社内Git、共有フォルダ等で配布するツール

## 必要な入力

最低限、次のいずれかを対象として受け取る。

- Goプロジェクトのルートディレクトリ
- `go.mod` を含むリポジトリ
- 配布予定バイナリと、そのソースリポジトリ

可能なら次も確認する。

- 想定利用者と用途
- 対応OS・CPUアーキテクチャ
- 配布方法
- 利用する社内・外部システム
- 外部通信先
- 扱うデータの機密区分
- 公開予定バージョン

情報が不足していても実施可能な検査を先に行い、不足情報は最終報告の「確認待ち」に列挙する。事実が確認できない項目を推測で合格にしない。

## 基本原則

1. リポジトリ内の `AGENTS.md`、`.github/copilot-instructions.md`、README、CONTRIBUTING、社内規程がある場合は最初に読む。
2. 読み取り中心で進め、破壊的コマンド、公開、push、タグ作成、依存関係更新を行わない。
3. シークレットの値を報告書へ転載しない。種類、ファイル、行番号だけを示し、値は必ずマスクする。
4. ツールが未インストール、ネットワーク利用不可、権限不足の場合は「未実施」とし、合格扱いにしない。
5. 既存ファイルを変更せず、必要な一時生成物は一時ディレクトリへ出力する。
6. `go test ./...` だけでなく、可能な限り `go vet`、脆弱性、ライセンス、クロスビルド、配布物を確認する。
7. 自動検査結果を鵜呑みにせず、コード、設定、用途との整合性を確認する。

## 判定基準

### FAIL

次のいずれかがある場合は公開不可とする。

- 認証情報、APIキー、秘密鍵、パスワード、個人情報等がリポジトリまたは配布物に含まれる
- 既知の重大または高リスク脆弱性があり、到達可能性または影響を否定できない
- ライセンス条件により想定する社内配布が認められない、または義務を満たせない
- ビルド不能、主要テスト失敗、起動不能
- 意図しない外部送信、危険な権限要求、任意コマンド実行等の重大リスクがある
- 必須の社内審査・承認が欠けている

### CONDITIONAL PASS

公開を直ちに止める問題はないが、公開前の対応または責任者判断が必要な場合。

- 中リスク以下の脆弱性または保守上の懸念
- 一部環境のテスト未実施
- SBOM、ライセンス一覧、操作手順、ロールバック手順等が不足
- 警告、低いテストカバレッジ、非推奨API等が残る

### PASS

必須検査が完了し、公開阻害事項がなく、配布・利用・障害対応に必要な情報が揃っている場合。

## 実施手順

### 1. 対象と公開条件を把握する

次を確認する。

- モジュール名、Goバージョン、主要コマンド
- 対応OS・CPU
- 入出力、ファイルアクセス、環境変数、設定ファイル
- ネットワーク通信先、プロキシ、TLS、認証方式
- 必要権限、管理者権限の要否
- ログの出力先と内容
- 配布物、インストール、更新、アンインストール方法

まず以下を調査する。

```bash
pwd
find . -maxdepth 3 -type f | sort
go env GOMOD GOVERSION GOOS GOARCH GOPROXY GOSUMDB GOPRIVATE
sed -n '1,240p' go.mod
```

`vendor`、生成ファイル、大規模ディレクトリは必要に応じて除外する。

### 2. リポジトリ衛生と機密情報を確認する

確認項目:

- 秘密鍵、トークン、接続文字列、資格情報
- `.env`、証明書、バックアップ、ダンプ、実データ
- 個人名、メールアドレス、端末固有パス、社内URL
- デバッグコード、テスト用認証情報、固定パス
- `.gitignore` の不足
- Git履歴に残った機密情報

利用可能なら専用スキャナーを優先する。

```bash
git status --short
git ls-files
gitleaks detect --source . --no-banner --redact
```

専用ツールがなければ、誤検知に注意しつつ文字列検索を補助的に行う。検出値そのものは出力しない。

### 3. 依存関係と再現性を確認する

```bash
go mod verify
go mod tidy -diff
go list -m -json all
go list -deps ./...
```

確認項目:

- `go.mod` と `go.sum` の整合
- 不要・不足依存
- `replace` / `exclude` / 擬似バージョンの妥当性
- 未公開・個人管理・ローカルパス依存
- `GOPROXY`、`GOSUMDB`、`GOPRIVATE` の社内方針適合
- ビルド時に外部ネットワークへ依存するか
- Go toolchainと依存バージョンの固定方針

`go mod tidy -diff` が利用できないGoバージョンでは、作業コピー上で差分を確認し、元ファイルは変更しない。

### 4. フォーマット、静的解析、テストを実行する

```bash
test -z "$(gofmt -l .)"
go vet ./...
go test ./...
go test -race ./...
go test -coverprofile=coverage.out ./...
go tool cover -func=coverage.out
```

利用可能なら追加する。

```bash
staticcheck ./...
golangci-lint run
```

注意:

- `-race` は対応環境とCGOを必要とする場合がある。未実施理由を記録する。
- カバレッジ率だけで合否を決めず、認証、権限、削除、上書き、外部通信、異常系等の重要経路を確認する。
- テストが外部システムや実データを変更しないことを確認してから実行する。

### 5. 脆弱性と危険な実装を確認する

```bash
govulncheck ./...
gosec ./...
```

目視確認項目:

- `os/exec`、シェル呼び出し、引数の連結
- SQL、テンプレート、パス、URLへの未検証入力
- パストラバーサル、Zip Slip、シンボリックリンク
- 不安全な一時ファイルと過剰なファイル権限
- TLS検証無効化、平文HTTP、弱い暗号、証明書処理
- SSRF、無制限リダイレクト、タイムアウトのないHTTP通信
- 認証・認可漏れ
- ログへの秘密情報・個人情報出力
- 無制限入力、巨大ファイル、無限ループ、goroutine漏れ
- `unsafe`、cgo、プラグイン、動的ロード
- エラーの握りつぶし、panic依存

検出結果は、コードで実際に到達可能か、入力が信頼境界を越えるかを確認して優先度を付ける。

### 6. ライセンスと知的財産を確認する

確認項目:

- リポジトリ自身のライセンスまたは社内利用条件
- 直接・間接依存のライセンス
- NOTICE、著作権表示、ソース提供等の義務
- GPL、AGPL、LGPL、SSPLその他の社内方針上注意すべき条件
- コピーペーストしたコード、生成コード、画像、辞書、サンプルデータ
- フォント、アイコン、外部バイナリ等、Go module外の資産

利用可能なライセンス検査ツールを使用し、その出力を目視確認する。

```bash
go-licenses report ./...
go-licenses check ./...
```

ライセンスの最終判断は法務・知財部門または社内規程に従う。このスキルだけで法的適合を断定しない。

### 7. ビルドと配布物を確認する

通常ビルド:

```bash
go build -trimpath ./...
```

CLI等の対象コマンドが明確な場合は、一時ディレクトリへ明示的に出力する。

```bash
mkdir -p .tmp-release-check
go build -trimpath -ldflags="-s -w" -o .tmp-release-check/tool ./cmd/tool
```

必要な対象だけクロスビルドする。

```bash
GOOS=windows GOARCH=amd64 go build -trimpath -o .tmp-release-check/tool-windows-amd64.exe ./cmd/tool
GOOS=linux GOARCH=amd64 go build -trimpath -o .tmp-release-check/tool-linux-amd64 ./cmd/tool
```

確認項目:

- 対象プラットフォームでビルドできる
- バージョン情報を表示できる
- `-trimpath` 等により開発端末の絶対パスが混入しない
- デバッグ情報、設定、秘密情報、不要ファイルを同梱しない
- バイナリ名、サイズ、チェックサムが記録される
- マルウェア対策製品での検査方法が定義される
- コード署名が必要な場合は署名手順がある
- アンインストールと更新・ロールバックが可能

```bash
sha256sum .tmp-release-check/*
```

Windowsのみの対象は、実際のWindows環境でも起動・主要操作を確認する。クロスコンパイル成功だけで動作保証とはしない。

### 8. SBOMと供給網を確認する

利用可能ならSBOMを生成する。

```bash
syft dir:. -o cyclonedx-json=.tmp-release-check/sbom.cdx.json
```

確認項目:

- GitHub Actions等のアクションをコミットSHAで固定しているか
- リリース操作に最小権限が設定されているか
- 依存更新のレビュー手順があるか
- ビルド元コミット、Goバージョン、ビルド日時、チェックサムを追跡できるか
- 生成物を信頼できる場所へ保管するか
- CIログや成果物に秘密情報が含まれないか

### 9. 運用性と利用者向け文書を確認する

READMEまたは運用文書に次があるか確認する。

- ツールの目的、対象利用者、想定外用途
- 対応環境、前提ソフト、必要権限
- インストール、初期設定、実行例
- 入力・出力、保存先、外部通信
- 設定値と環境変数。秘密情報の安全な渡し方
- エラー時の対処、ログ確認方法
- 更新、ロールバック、アンインストール
- 問い合わせ先、保守責任者、サポート期限
- 既知の制約
- ライセンス、NOTICE、SBOMの所在

### 10. 最終判定を作成する

重要度を次の4段階で分類する。

- `BLOCKER`: 公開不可。秘密漏えい、重大脆弱性、違法・規程違反、主要機能不能
- `HIGH`: 原則として公開前に修正。悪用可能な欠陥、重要な運用欠落
- `MEDIUM`: 公開前対応または責任者のリスク受容が必要
- `LOW`: 改善推奨

同じ根本原因の指摘はまとめ、件数を水増ししない。各指摘には、証拠、影響、再現または確認方法、推奨対応を含める。

## 最終報告書の形式

以下の形式で日本語のMarkdownを出力する。

```markdown
# Goツール 社内公開前チェック結果

## 1. 結論

- 判定: PASS / CONDITIONAL PASS / FAIL
- 対象: <リポジトリ・モジュール・バージョン>
- 対応環境: <OS/ARCH>
- 検査日時: <日時>
- 検査対象コミット: <commit SHA>
- 公開阻害事項: <件数>
- 公開前対応事項: <件数>

## 2. 判定理由

<3～8行で要約>

## 3. 指摘事項

### [BLOCKER-001] <件名>

- 分類: Security / License / Privacy / Quality / Operations / Distribution
- 証拠: `<ファイル>:<行>` またはコマンド結果
- 内容: <確認できた事実>
- 影響: <社内公開時の影響>
- 推奨対応: <具体的な対応>
- 再確認方法: <修正後の確認方法>

## 4. チェック結果

- [x] 実施済み項目
- [ ] 未実施項目 — 理由: <理由>
- [!] 要対応項目 — <概要>

## 5. 配布物

- ファイル名:
- 対象OS/ARCH:
- SHA-256:
- ビルド元コミット:
- Goバージョン:
- SBOM:
- 署名:

## 6. 確認待ち

- <責任者または利用者へ確認すべき事項>

## 7. 実行したコマンド

<秘密情報を含まないコマンドのみ>

## 8. 公開前アクション

1. <担当・期限・完了条件が明確な対応>
```

## 出力時のルール

- 問題が見つからなくても、未実施検査と残余リスクを明記する。
- コマンド出力を全文貼り付けず、判断に必要な部分を要約する。
- シークレット候補は `****` に置き換える。
- ファイルと行番号を可能な限り示す。
- 法務判断、脆弱性の悪用可能性、社内規程適合を推測で断定しない。
- `PASS` は必須検査を実施できた場合だけ使用する。
- 修正案は最小限かつ具体的にし、依頼なしにファイルを変更しない。

## 中止条件

次の場合は該当操作を止め、理由と安全な代替案を報告する。

- 本番システム、実データ、実アカウントへ影響する可能性がある
- 破壊的処理、権限昇格、公開、push、リリース操作が必要
- 機密情報を外部サービスへ送信する必要がある
- ライセンスや規程上の判断に権限者の確認が必要
- マルウェアと思われる挙動または意図不明な難読化コードを検出した
